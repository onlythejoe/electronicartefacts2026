# Repository Architecture

This repo is a static site with a typed knowledge-graph build layer. The public website is committed as static output, but the source of truth is the structured content, templates, config, and build code.

## Architectural Layers

| Layer | Paths | Responsibility |
| --- | --- | --- |
| Content source | `content/**/*.md`, `content/relations/*.yaml` | Canonical entities and typed relation statements. |
| Legacy compatibility input | `src/legacy-data/` | Historical browser-runtime data; kept outside the published asset tree. |
| Contracts | `src/schema/`, `src/config/` | Entity types, relation predicates, route rules, publication contracts, and site constants. |
| Build pipeline | `src/build/`, `scripts/build-site.ts`, `scripts/validate-content.ts` | Load content, validate graph integrity, build routes, render output, and write manifests. |
| Rendering | `src/templates/`, `assets/partials/` | HTML layout, page templates, reusable components, header, and footer. |
| Derived data | `src/graph/`, `src/search/`, `src/seo/`, `src/semantic/` | Graph neighborhoods, search documents, metadata, sitemap, and JSON-LD. |
| Static assets | `assets/css/`, `assets/js/`, `assets/media/`, `assets/img/` | Browser runtime, styling, media, partials, and other public assets. |
| Published output | root `*.html`, route folders, `generated/`, `graph/`, `id/`, `search/`, `sitemap.xml` | Committed static site output served by the host. |
| Verification | `tests/`, `npm run check` | Regression coverage for schemas, routes, graph, search, SEO, and generated output. |
| Documentation | `docs/` | Internal architecture notes, audits, and implementation records. |

## Source And Generated Boundaries

Use `content/`, `src/`, `scripts/`, `assets/partials/`, and shared assets as the main editing surface.

Treat generated entity pages, identifier routes, search indexes, graph views, manifests, and sitemap files as build output. If generated output is wrong, fix the content, schema, template, or build code first, then run `npm run build`.

Root hand-authored public pages can still be edited directly when they are not generated by `scripts/build-site.ts`. Keep generated and hand-authored pages conceptually separate during reviews.

## Change Rules

Add a new entity by creating a typed Markdown file under the matching `content/` folder, adding any relation statements in `content/relations/`, then running `npm run validate` and `npm run build`.

Add a new entity type by updating the schema, route rules, metadata/JSON-LD behavior, templates if needed, and focused tests before generating output.

Add a new public route by deciding first whether it is content-generated or hand-authored. Generated routes belong in `src/config/routes.ts` and the build pipeline; hand-authored public pages should stay visible in the published static surface.

Add new frontend behavior under `assets/js/` and keep data-only generated artifacts under `generated/`, `graph/`, or `search/`.

Before shipping architecture changes, run:

```sh
npm run validate
npm run typecheck
npm test
```

Run `npm run build` when content, templates, SEO, graph, search, or route behavior changes.
